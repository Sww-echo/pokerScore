# PokerScore H5 需求完善与实现方案

## 1. 项目目标

PokerScore 是一个用于线下打牌场景的 H5 记分工具。

核心目标：

- 用“分”代替“钱”记录输赢变化
- 通过房间内实时划转分值，替代人工口算
- 在牌局结束后生成清晰的最终结算结果
- 结算仅作为线下转账参考，不直接处理线上支付

## 2. 已确认的业务决策

- 不限定必须在微信中使用
- 非微信浏览器访问时，允许用户输入昵称后进入
- 也允许系统生成默认昵称作为兜底
- 允许游客进入房间
- 允许分数出现负数
- 房间内任意成员都可以点击结算
- 结算后允许重新开启房间并继续记分

## 3. 产品范围定义

### 3.1 V1 必做能力

- 用户登录或游客进入
- 创建房间
- 输入房间码进入房间
- 扫码进入房间
- 房间成员展示
- 单人转分
- 多人转分
- 实时同步成员分数变化
- 实时同步转分流水
- 结算页查看最终分数
- 自动生成建议转账方案

### 3.2 V2 可选增强

- 最近一笔转分撤销
- 房主权限控制
- 房间历史记录
- 结算截图导出
- 成员备注名
- 房间规则模板

## 4. 用户角色

### 4.1 游客用户

- 输入昵称后进入房间
- 使用默认头像与默认昵称
- 可以参与转分和结算

### 4.2 微信登录用户

- 使用微信头像和昵称
- 在微信环境内体验更自然
- 后续可扩展用户历史房间和结算记录

## 5. 核心业务规则

### 5.1 房间规则

- 房间由任意用户创建
- 房间生成唯一房间码
- 房间支持二维码分享
- 房间状态包含：`active`、`settled`
- 结算后允许重新开启，重新开启后状态回到 `active`

### 5.2 成员规则

- 成员进入房间后展示头像、昵称、当前净分
- 游客头像使用系统默认头像
- 昵称在房间中必须可识别，必要时自动追加编号避免重名
- 用户刷新页面或断线重连后，应恢复当前房间状态

### 5.3 分数规则

- 分值使用整数
- 允许负数
- 初始分数为 `0`
- 只记录房间成员之间的转移
- 房间内所有成员分数总和恒为 `0`
- 单次转分必须大于 `0`
- 一次多人转分，本质上拆成多条转分流水

### 5.4 转分规则

- 支持 `A -> B`
- 支持 `A -> B/C/D`
- 多人转分时，每个接收方单独输入分值
- 提交前需二次确认
- 成功后房间所有成员实时看到最新分数和流水

### 5.5 结算规则

- 任意成员均可点击结算
- 点击结算时需要二次确认
- 结算页展示每个成员最终净分、排名信息、建议转账方案
- 建议转账方案目标是减少实际线下转账笔数
- 结算后允许继续开启房间并继续记分

## 6. 页面设计

### 6.1 页面列表

V1 建议包含 5 个核心页面：

1. 进入页
2. 登录或昵称确认弹窗
3. 创建或加入房间页
4. 房间页
5. 结算页

### 6.2 进入页

模块建议：

- 产品标题
- 产品说明文案
- `创建房间` 主按钮
- `加入房间` 主按钮
- 房间码输入入口
- 扫码入口
- 微信登录按钮
- 游客进入按钮

建议文案：

- 标题：`PokerScore`
- 副标题：`用分代替钱，实时记牌局输赢`

### 6.3 登录或昵称确认弹窗

触发场景：

- 非微信浏览器访问
- 微信授权失败
- 用户主动选择游客进入

字段建议：

- 昵称输入框
- 默认昵称按钮
- 确认进入按钮

### 6.4 创建或加入房间页

创建房间字段建议：

- 房间名称，可选
- 房间人数上限，可选，默认 10
- 创建按钮

加入房间字段建议：

- 房间码输入框
- 加入按钮
- 扫码按钮

### 6.5 房间页

页面结构：

- 顶部区：房间名称、房间码、分享按钮、二维码按钮、在线人数
- 中部区：成员卡片列表，每张卡片显示头像、昵称、当前分数
- 操作区：单人转分按钮、多人转分按钮、结算按钮
- 底部区：实时流水列表

成员卡片交互：

- 点击任意成员头像或卡片，弹出转分面板
- 当前用户不可给自己转分

### 6.6 转分弹窗

弹窗字段建议：

- 付款方
- 收款方选择区
- 每个收款方对应分值输入框
- 本次总转出分值
- 确认按钮
- 取消按钮

校验规则：

- 金额必须为正整数
- 不允许选中自己
- 至少选择一个收款方
- 至少填写一个有效金额

### 6.7 结算页

页面结构：

- 顶部：房间名称、结算时间
- 最终分数区：成员头像、昵称、最终净分
- 建议转账区：谁转给谁多少
- 操作区：重新开启房间、返回房间

示例：

- A `+100`
- B `-60`
- C `-40`

建议转账方案：

- B 转 A 60
- C 转 A 40

## 7. 实时同步需求

需要实时同步以下事件：

- 成员进入房间
- 成员离开房间
- 分数划转成功
- 房间结算
- 房间重新开启
- 成员昵称或头像更新

## 10. 技术方案选项

### 10.1 方案 A：Vue 3 H5 + Node.js + WebSocket

技术栈建议：

- 前端：`Vue 3` + `Vite` + `Pinia` + `Vant`
- 后端：`NestJS` 或 `Express`
- 实时通信：`Socket.IO`
- 数据库：`MySQL`
- 缓存：`Redis`，可选

优点：贴合 H5 场景，实时能力直接，后续扩展顺畅。

缺点：需要独立后端服务。

### 10.2 方案 B：uni-app + Node.js

优点：后续转微信小程序更方便。

缺点：纯 H5 体验和开发效率通常不如 Vue 原生方案。

### 10.3 方案 C：云开发 MVP

优点：上线快。

缺点：复杂实时房间和后续扩展空间有限。

## 11. 推荐方案

推荐优先选择方案 A。

原因：

- 当前目标是 H5，不是小程序优先
- 核心问题是多人房间实时同步，`WebSocket + Node.js` 更稳妥
- 后续扩展游客、微信登录、历史记录、撤销、结算快照更顺畅

建议最小组合：

- 前端：`Vue 3 + Vite + Pinia + Vant`
- 后端：`NestJS`
- 实时通信：`Socket.IO`
- 数据库：`MySQL`

## 13. 页面视觉建议

- 不是金融记账风，而是轻牌桌工具风
- 主色推荐墨绿、深木色、米白
- 成员区强调头像和当前分数
- 正分用暖色，负分用冷色
- 流水区信息密度高但样式克制
