# PokerScore 用户体系升级方案

适用版本：

- 当前项目阶段：游客模式为主
- 当前日期：`2026-03-08`
- 本文目标：先明确后续实现方案，不在本次落地代码

## 1. 背景

当前系统的用户逻辑是：

- 前端默认使用 `guest` 模式
- 浏览器本地生成并保存 `deviceId`
- 后端用 `deviceId` 识别或复用游客用户
- 用户资料、房间状态、token 保存在本地 `localStorage`

当前实现的优点：

- 首次进入门槛低
- 无需接入第三方短信或微信
- 创建房间、加入房间流程足够轻

当前实现的主要问题：

- 清理浏览器缓存后，用户身份可能丢失
- 更换设备后，无法恢复原来的用户身份
- 无法做到“旧设备确认后新设备直接登录”
- 现有用户体系更接近“设备身份”，不是真正可恢复的“用户身份”

## 2. 方案目标

在不依赖第三方短信、微信登录的前提下，升级为一套“可恢复、可迁移、依然低门槛”的用户体系。

目标：

- 保留游客模式，保证首屏转化
- 提供恢复码，支持旧设备不在场时恢复账号
- 提供旧设备扫码登录，支持设备迁移
- 保证现有房间、成员、转分、结算逻辑尽量少改

非目标：

- 不做实名体系
- 不做手机号验证码
- 不做真实微信 OAuth
- 不在本阶段实现复杂的账号密码体系

## 3. 目标用户流程

### 3.1 首次进入

用户第一次进入时，仍然按游客模式使用：

1. 输入昵称或使用随机昵称
2. 创建房间或加入房间
3. 后端创建或复用游客用户
4. 后端签发 token / refresh token
5. 前端保存本地状态

### 3.2 保存恢复码

在以下时机提示用户保存恢复码：

- 第一次创建房间成功后
- 第一次加入房间成功后
- 第一次产生转分记录后
- 用户主动进入“账号与设备”页面时

恢复码的作用：

- 旧设备丢失或不可用时，用户可在新设备输入恢复码恢复身份

### 3.3 使用恢复码恢复账号

当用户更换设备且旧设备不可用时：

1. 新设备点击“恢复账号”
2. 输入恢复码
3. 后端校验恢复码
4. 校验通过后，为新设备创建会话并返回 token
5. 前端进入已恢复的原账号状态

### 3.4 旧设备扫码登录新设备

当用户旧设备还在手上时：

1. 新设备点击“使用旧设备扫码登录”
2. 后端创建一个短期有效的登录挑战
3. 新设备展示二维码
4. 旧设备扫码识别挑战
5. 旧设备展示确认页，用户点击确认
6. 后端把新设备登录到当前用户
7. 新设备轮询或实时收到登录成功状态

## 4. 方案核心设计

### 4.1 核心原则

- 用户身份和设备身份分离
- 游客模式继续保留
- 恢复能力通过恢复码补齐
- 迁移能力通过旧设备扫码授权补齐
- 后端开始显式管理“用户会话”

### 4.2 核心对象

#### A. 用户

表示长期身份，依然以现有 `app_user` 为核心。

#### B. 设备会话

表示“某个用户在某台设备上的登录状态”。

#### C. 恢复码

表示“用户可脱离旧设备找回账号”的凭证。

#### D. 登录挑战

表示“旧设备授权新设备登录”的临时凭证。

## 5. 建议的数据模型

当前数据库已有：

- `app_user`
- `room`
- `room_member`
- `room_round`
- `transfer_batch`
- `transfer_record`
- `settlement_snapshot`
- `room_event`

后续建议新增以下表。

### 5.1 `user_session`

用途：

- 记录多设备登录状态
- 支持 refresh token 续期
- 支持主动失效、会话管理

建议字段：

- `id`
- `user_id`
- `device_id`
- `refresh_token_hash`
- `user_agent`
- `ip_address`
- `created_at`
- `last_seen_at`
- `revoked_at`

说明：

- `refresh_token` 不应明文存储，只存 hash
- 同一用户可有多个 session

### 5.2 `user_recovery_code`

用途：

- 支持新设备通过恢复码恢复身份

建议字段：

- `id`
- `user_id`
- `code_hash`
- `code_last4`
- `status`
- `created_at`
- `used_at`
- `revoked_at`

状态建议：

- `active`
- `used`
- `revoked`

说明：

- 恢复码只展示一次明文，数据库只存 hash
- `code_last4` 用于前端展示和审计，不用于校验

### 5.3 `device_login_challenge`

用途：

- 支持旧设备扫码授权新设备登录

建议字段：

- `id`
- `challenge_token`
- `target_device_id`
- `request_user_agent`
- `request_ip`
- `approved_by_user_id`
- `status`
- `expires_at`
- `approved_at`
- `used_at`
- `rejected_at`

状态建议：

- `pending`
- `approved`
- `rejected`
- `expired`
- `used`

说明：

- 挑战必须短时有效
- 每个 challenge 只能消费一次

## 6. 对现有 `app_user` 的建议补充

建议补充以下字段：

- `account_level`
- `last_login_at`
- `last_seen_at`

其中：

- `account_level` 可选值：`guest`、`recoverable`

说明：

- 当前所有用户默认都是 `guest`
- 当用户成功生成并保存恢复码后，可升级为 `recoverable`

## 7. 接口设计建议

以下为建议接口，不要求与当前 `openapi-v1.yaml` 完全一致，可作为下阶段增量设计。

### 7.1 继续保留

- `POST /api/v1/auth/guest`

用途：

- 游客认证，创建或复用游客用户

### 7.2 恢复码接口

#### `POST /api/v1/auth/recovery-codes`

用途：

- 为当前登录用户生成恢复码

返回：

- 明文恢复码，仅返回一次
- 可附带 `codeLast4`

#### `POST /api/v1/auth/recovery-codes/rotate`

用途：

- 用户主动重置恢复码

返回：

- 新恢复码
- 旧恢复码立即失效

#### `POST /api/v1/auth/recovery-codes/restore`

用途：

- 用恢复码在新设备恢复账号

入参建议：

- `recoveryCode`
- `deviceId`

返回：

- `user`
- `token`
- `refreshToken`

### 7.3 旧设备扫码登录接口

#### `POST /api/v1/auth/device-login/challenges`

用途：

- 新设备发起登录挑战

入参建议：

- `deviceId`

返回：

- `challengeId`
- `challengeToken`
- `expiresAt`
- 用于生成二维码的 payload

#### `GET /api/v1/auth/device-login/challenges/:challengeId`

用途：

- 新设备轮询挑战状态

返回：

- `pending`
- `approved`
- `rejected`
- `expired`

若已通过：

- 返回新的登录凭证或返回一个可换 token 的临时授权码

#### `POST /api/v1/auth/device-login/challenges/:challengeId/approve`

用途：

- 旧设备确认授权新设备登录

要求：

- 当前旧设备必须已登录

#### `POST /api/v1/auth/device-login/challenges/:challengeId/reject`

用途：

- 旧设备拒绝授权

## 8. 前端交互建议

### 8.1 首页新增入口

建议在首页补两个入口：

- `恢复账号`
- `旧设备扫码登录`

要求：

- 入口不要干扰创建房间和加入房间主路径
- 可放在登录方式区域下方，或放在二级入口中

### 8.2 房间页/个人入口

建议后续增加“账号与设备”页，至少包含：

- 当前身份状态
- 是否已生成恢复码
- 重新生成恢复码
- 当前设备信息
- 其他设备会话列表

### 8.3 二维码识别复用

项目里已有扫码能力，可复用当前扫码组件：

- [QrScannerModal.vue](../src/components/QrScannerModal.vue)

后续扫码内容需要支持两类 payload：

- 房间邀请码
- 设备登录挑战码

建议二维码 payload 格式做前缀区分，例如：

- `pokerscore://room?code=ABCD12`
- `pokerscore://device-login?challenge=xxx`

## 9. 安全要求

### 9.1 恢复码

- 恢复码必须只存 hash
- 必须限制尝试次数
- 必须做接口限流
- 建议使用后自动轮换
- 建议在生成成功后明确提示“请妥善保存”

### 9.2 扫码登录

- challenge 必须有过期时间，建议 `2~5` 分钟
- challenge 只能使用一次
- 旧设备必须明确确认
- 前端要展示新设备基础信息，减少误授权

### 9.3 会话管理

- refresh token 只存 hash
- 支持单设备失效
- 支持全设备退出
- 关键操作建议记录审计日志

## 10. 分阶段实施建议

### Phase 1

目标：

- 保持现有游客模式不变
- 新增恢复码能力

实施内容：

- 新增 `user_recovery_code`
- 新增恢复码生成与恢复接口
- 首页增加“恢复账号”入口

收益：

- 先解决“换设备后找不回”的核心问题

### Phase 2

目标：

- 引入正式的会话管理

实施内容：

- 新增 `user_session`
- refresh token 改为后端可追踪
- 支持基础会话管理

收益：

- 用户体系从“本地 token”升级为“后端可管理会话”

### Phase 3

目标：

- 新增旧设备扫码登录

实施内容：

- 新增 `device_login_challenge`
- 新设备创建挑战
- 旧设备扫码确认
- 新设备自动登录

收益：

- 设备迁移体验明显提升

## 11. 推荐实现顺序

推荐顺序：

1. 先做恢复码
2. 再做会话表
3. 最后做扫码登录

原因：

- 恢复码是最小闭环
- 对当前系统侵入最小
- 扫码登录体验更好，但复杂度更高

## 12. 当前实现时的注意事项

### 12.1 不建议直接移除游客模式

原因：

- 会明显提高进入门槛
- 不符合当前项目的工具型产品定位

### 12.2 不建议直接上账号密码

原因：

- 忘记密码后仍需要找回机制
- 对当前产品价值来说过重

### 12.3 不建议继续只靠 `deviceId`

原因：

- 清缓存或换设备就会丢失身份
- 无法满足“真实可恢复用户”的目标

## 14. 结论

`游客模式 + 恢复码 + 旧设备扫码登录` 是当前阶段最适合 PokerScore 的用户体系升级方案。

它的优势是：

- 无需接入第三方短信或微信
- 保留首屏低门槛体验
- 能支持账号恢复
- 能支持跨设备迁移
- 能在现有游客体系基础上逐步演进

建议下一步先落地恢复码，再逐步补齐会话管理和旧设备扫码登录。
